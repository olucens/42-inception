# Docker — глубокий чит-лист (мой формат)

Не про "запомнить команду", а про "понять, почему именно так".
Проверка на прочность: удалить `Inception/srcs` и написать заново по памяти, глядя только сюда.

> Полный список голых CLI-команд (`docker ps`, `docker logs`, ...) уже есть в [cheatsheet.md](cheatsheet.md) — здесь он не дублируется.

---

## 1. Под капотом (по шагам, просто)

**Одна фраза, которая всё объясняет:** Docker = удобная обёртка над возможностями ядра Linux. Магии нет.

### 🧩 Процессы и ресурсы

**PID namespace — контейнер сам себе мир**
- Что: внутри контейнера первый процесс всегда PID 1, как будто он один в системе.
- Руками:
  ```bash
  unshare --pid --fork chroot ./rootfs /bin/sh
  ```
- Флаг `--fork` — обязателен. `unshare` меняет namespace только для процессов, созданных **после** вызова, поэтому нужен fork нового процесса.
- Зачем помнить: PID 1 — единственный, кто получает `SIGTERM` от `docker stop`. Не ответил за 10 сек → `SIGKILL`.
- → Inception: поэтому `bash`, `tail -f`, `sleep infinity`, `while true` запрещены как entrypoint — это не процесс, а затычка, которая не реагирует на сигналы правильно.

**cgroups — лимит ресурсов**
- Что: не даёт одному контейнеру сожрать всю RAM/CPU хоста.
- Руками:
  ```bash
  mkdir /sys/fs/cgroup/x
  echo 512M > /sys/fs/cgroup/x/memory.max
  echo $$ > /sys/fs/cgroup/x/cgroup.procs
  ```
- Compose-эквивалент: `deploy.resources.limits.memory`.
- Зачем помнить: без лимита один упавший в утечку контейнер роняет весь хост через OOM killer.

### 📁 Файловая система

**OverlayFS — слои образа**
- Что: образ = стопка read-only слоёв + один read-write слой поверх.
- Руками:
  ```bash
  mount -t overlay overlay -o lowerdir=L2:L1,upperdir=U,workdir=W merged/
  ```
- Деталь: в `lowerdir` первый в списке — самый "верхний" по приоритету среди read-only слоёв.
- Зачем помнить: правка 1 байта в файле 100 МБ копирует все 100 МБ в верхний слой (Copy-on-Write). Большие изменяемые данные (БД) → в volume, не в тело образа.

**Именованный volume**
- Что: данные, которые переживают `docker rm`.
- Руками:
  ```bash
  mount --bind /var/lib/docker/volumes/name/_data /var/lib/mysql
  ```
- Зачем помнить: монтируется **поверх** уже готового OverlayFS-view, отдельно от слоёв образа → не участвует в CoW, не исчезает вместе с контейнером.

### 🌐 Сеть

**Bridge + veth — как контейнер вообще выходит в сеть**
- Что: виртуальный "патч-корд" между хостом и контейнером.
- Руками:
  ```bash
  ip link add veth0 type veth peer name veth1
  ip link set veth1 netns $PID
  ip link set veth0 master docker0
  ```
- Зачем помнить: без veth-пары у изолированного network namespace вообще нет пути наружу.

**DNS по имени сервиса**
- Что: `ping wordpress` работает, потому что `dockerd` держит DNS-резолвер на `127.0.0.11` внутри **кастомной** сети.
- Руками (без этого): пришлось бы вручную дописывать `/etc/hosts` в каждый контейнер при каждом старте.
- → Inception: на default bridge такого DNS нет — раньше связывали через `--link`. Кастомная сеть — современная замена, поэтому `--link`/`links:` запрещены.

**Публикация порта `-p 443:443`**
- Что: правило NAT, связывающее порт хоста с портом контейнера.
- Руками:
  ```bash
  iptables -t nat -A DOCKER -p tcp --dport 443 -j DNAT --to-destination 172.x.x.x:443
  ```
- Зачем помнить: без этого правила хостовый порт и порт внутри netns — два никак не связанных пространства.

### 🔑 Секреты и сборка

**Docker secret**
- Что: файл с паролем монтируется в контейнер, а не передаётся через переменную окружения.
- Руками:
  ```bash
  mount --bind -o ro /path/secret_file /run/secrets/db_password
  ```
- Зачем помнить: в отличие от `ENV`/`environment:`, не виден в `docker inspect → Config.Env`, не попадает в слои образа. Поэтому именно secrets, а не переменные, для паролей.

**`docker build` и кэш**
- Что: каждая инструкция Dockerfile = отдельный слой + отдельная запись в кэше.
- Руками: `chroot rootfs sh -c "<RUN>"` → снять diff → сохранить как слой, повторить на каждую инструкцию.
- Зачем помнить: кэш ломается по хэшу (инструкция + состояние предыдущего слоя). Одна изменившаяся `COPY` инвалидирует **всё**, что идёт ниже неё в файле.

### 🎛 Compose — просто оркестратор

- Что делает `docker compose up` на самом деле:
  ```bash
  docker network create ...
  docker volume create ...
  docker build ...        # ×N сервисов
  docker run --network --volume --env-file -d ...   # ×N, с учётом depends_on
  ```
- Зачем нужен именно Compose, а не bash-скрипт: `depends_on: condition: service_healthy` умеет ждать реальной готовности сервиса, а не просто порядка запуска.

## 2. Структура Dockerfile и docker-compose.yml

### 2a. Dockerfile — официальный список ВСЕХ инструкций + логика порядка

Порядок в Dockerfile **функционально значим** (влияет на кэш и итоговые слои). Ниже — весь официальный список, сгруппированный в том порядке, в котором инструкции обычно идут в реальном файле.

**Группа A — база (всегда в начале)**

| Инструкция | Что делает | Почему здесь |
|---|---|---|
| `ARG` (до `FROM`) | Билд-переменная, доступна только до `FROM`, если нужно параметризовать сам базовый образ (`FROM debian:$VER`) | Единственное место, где `ARG` вообще может стоять раньше `FROM` |
| `FROM base:tag [AS name]` | Задаёт базовый слой | Всегда первая "настоящая" инструкция — всё остальное строится поверх |

**Группа B — метаданные и подготовка окружения**

| Инструкция | Что делает | Почему здесь |
|---|---|---|
| `LABEL` | Метаданные образа (автор, версия) | Дёшево и не меняется — рано, чтобы не сбрасывать кэш |
| `ENV` | Переменные окружения, доступны и на build, и в рантайме | Ставится до `RUN`, если нужна внутри установки (`DEBIAN_FRONTEND=noninteractive`) |
| `ARG` (после `FROM`) | Билд-переменная, живёт только во время сборки, в рантайм не попадает | Нужен для build-time настроек, которые не должны утечь в финальный образ |

**Группа C — установка зависимостей (редко меняется → рано → кэшируется)**

| Инструкция | Что делает | Почему здесь |
|---|---|---|
| `RUN` | Выполняет команду и коммитит результат новым слоем | `apt-get update && install && rm -rf /var/lib/apt/lists/*` — всё **в одной** инструкции, иначе очистка кэша apt ничего не даст (предыдущий слой уже зафиксирован) |
| `WORKDIR` | Устанавливает рабочую директорию (создаёт, если её нет) | До `COPY`, чтобы относительные пути назначения были чистыми, без `cd` внутри `RUN` |

**Группа D — файлы приложения (меняются часто → позже → чаще ломают кэш)**

| Инструкция | Что делает | Почему здесь |
|---|---|---|
| `COPY` | Копирует файлы из контекста сборки в образ | Предпочтительнее `ADD` — предсказуемое поведение, без скрытой магии |
| `ADD` | То же самое + автораспаковка `.tar` + скачивание по URL | Использовать **только** если реально нужна эта магия — иначе непредсказуемое поведение (лучше `COPY`) |
| `RUN` (повторно) | Пост-обработка скопированного: `chmod +x`, компиляция и т.п. | Не полагаться на права файла с хоста (git не всегда сохраняет exec-бит) — `chmod +x` явно в Dockerfile |

**Группа E — рантайм-гигиена (безопасность/поведение контейнера)**

| Инструкция | Что делает | Почему здесь |
|---|---|---|
| `USER` | Переключает, от чьего имени выполняются следующие инструкции и финальный процесс | После всех установок (которым нужен root), перед `ENTRYPOINT` — принцип наименьших привилегий |
| `VOLUME` | Объявляет точку монтирования (если не переопределена в compose — создаётся анонимный volume) | Редко нужен явно в Inception — volume уже объявлен в `docker-compose.yml` |
| `EXPOSE` | **Только документирует** порт, ничего не публикует | Нет никакой `PORT`-инструкции — путают именно с `EXPOSE`. Реальная публикация — `ports:`/`expose:` в compose или `-p` в `docker run` |
| `STOPSIGNAL` | Меняет сигнал, который шлёт `docker stop` (по умолчанию `SIGTERM`) | Нужен, только если процесс ожидает нестандартный сигнал (например, nginx исторически различает `SIGTERM`/`SIGQUIT`) |
| `HEALTHCHECK` | Команда, которой Docker периодически проверяет "жив ли сервис по-настоящему" | **Критично для Inception**: `depends_on: condition: service_healthy` в compose работает только если у сервиса есть `HEALTHCHECK` |

**Группа F — финальный процесс (всегда в конце)**

| Инструкция | Что делает | Почему здесь |
|---|---|---|
| `CMD` | Команда по умолчанию: либо весь запуск, либо аргументы к `ENTRYPOINT` | Если задан и `ENTRYPOINT`, и `CMD` в exec-форме — `CMD` становится аргументами `ENTRYPOINT`. Если инструкция повторяется несколько раз в файле — считается **только последняя** |
| `ENTRYPOINT` | Фиксированный исполняемый файл контейнера | exec-форма `["init.sh"]` — процесс = PID 1 напрямую. Shell-форма `CMD nginx ...` без скобок заворачивает в `/bin/sh -c` → PID 1 = sh, а не nginx → `SIGTERM` не долетает до реального процесса |

**Не понадобится в Inception, но должен знать что это (официальный список полный)**

| Инструкция | Что делает | Почему не нужна здесь |
|---|---|---|
| `ONBUILD` | Регистрирует инструкцию, которая сработает, когда **этот** образ станет базовым (`FROM`) для другого Dockerfile | Сервисы Inception — конечные образы, их никто не наследует |
| `SHELL` | Меняет shell по умолчанию для shell-формы инструкций (по умолчанию `["/bin/sh","-c"]`) | Нужен только если пишешь bash-специфичный синтаксис в `RUN`/`CMD` |
| `MAINTAINER` | Старый способ указать автора образа | **Deprecated** — используй `LABEL maintainer="..."` |
| `COPY --from=stage` (multi-stage) | Копирует файл из промежуточного этапа сборки в финальный, более лёгкий образ | Полезно, когда есть шаг компиляции. В Inception все сервисы ставятся через `apt-get`, компилировать нечего |

### 2b. docker-compose.yml — порядок ключей **НЕ значим**

Важное отличие от Dockerfile: YAML — это словарь, парсеру всё равно, что выше что ниже. Порядок ниже — конвенция читаемости, не механика.

| Ключ | Что делает | Параметры / трейдофы | Правило / причина Inception |
|---|---|---|---|
| `build:` vs `image:` | `build` — собрать локально из своего Dockerfile; `image:` — просто имя для собранного образа (не pull!) | Если написать только `image: mariadb:latest` **без** `build:` — Docker попытается **стянуть** готовый образ с Hub | Готовые образы (`wordpress:`, `nginx:`, `mariadb:` из Docker Hub) **запрещены** — обязателен `build:` из собственного Dockerfile |
| `container_name:` | Фиксированное имя вместо авто-сгенерированного | Плюс: предсказуемое DNS-имя и `docker exec -it name`. Минус: несовместимо с `--scale` | Не проблема — по subject нужен ровно один инстанс на сервис |
| `env_file: .env` | Подставляет переменные из файла в окружение контейнера | Хранит несекретные настройки (имена БД, домен) вне самого compose-файла | `.env` **обязателен**, но **не для паролей** |
| `secrets:` | Монтирует файл как `/run/secrets/<name>`, read-only, не как env-переменную | Не светится в `docker inspect → Config.Env`, не попадает в логи | Пароли обязаны идти через secrets-файлы, **не через Dockerfile/ENV/committed-файлы** |
| `volumes:` (сервис) | `name:/path` монтирует именованный volume внутрь контейнера | `- /host/path:/path` (bind mount) технически тоже работает | **Bind mount для основных данных запрещён** — только именованный volume |
| `volumes:` (top-level) с `driver_opts: {type: none, o: bind, device: /home/.../data/mariadb}` | Именованный volume, физически лежащий по конкретному пути на хосте | Закрывает два требования: "именованный volume" + "данные в `/home/login/data`" | Каталог на хосте должен существовать **до** `compose up` — создаётся в Makefile |
| `networks:` (top-level), `driver: bridge` | Кастомная bridge-сеть вместо default | DNS по имени сервиса "из коробки" | `network: host` и `--link`/`links:` **запрещены** — замена: DNS кастомной сети |
| `restart: on-failure` | Перезапуск только при ненулевом exit-коде | `no` / `always` (даже после ручного stop) / `unless-stopped` | Именно `on-failure` — иначе намеренная остановка при проверке ревьюером выглядит как баг |
| `depends_on:` | Порядок **старта**, не готовности | `mysqld` может быть ещё не готов, когда wordpress уже стартовал | Для реальной готовности — `healthcheck:` + `condition: service_healthy` (см. `HEALTHCHECK` выше) |
| `expose:` vs `ports:` | `expose` — виден только другим контейнерам сети; `ports:` — публикует на хост | Только у nginx должен быть `ports: ["443:443"]` | mariadb/wordpress — только `expose`, NGINX обязана быть единственной точкой входа |

## 3. Что реально понадобится на защите

Не общий список CLI-команд (он в [cheatsheet.md](cheatsheet.md)), а конкретно: что спросит ревьюер → какая команда это доказывает.

| Что проверяют | Команда | Что она доказывает |
|---|---|---|
| Все сервисы подняты и настроены на restart | `docker compose ps` | Статус + restart policy |
| Не используются готовые образы | `docker images` → `docker history <image>` | Образ реально собран из debian/alpine, а не притянут с Hub |
| Нет `--link` / `network: host` | `docker network inspect inception-network` | driver: bridge, нет host-режима |
| Данные — в volume, не в контейнере | `docker volume inspect mariadb-data` | `Mountpoint` указывает на `/home/.../data/mariadb` |
| Пароли не в Dockerfile/git | `docker inspect mariadb --format '{{.Config.Env}}'` (пароля быть не должно) + `docker exec mariadb ls /run/secrets/` (файлы есть) | Пароль пришёл через secret-файл, а не через ENV |
| Контейнер переживает падение | `docker kill mariadb` → `docker compose ps` | Автоматически поднялся снова (restart: on-failure сработал) |
| NGINX — единственная точка входа, TLS 1.2/1.3 | `curl -vk https://akuzmin.42.fr` + `openssl s_client -connect host:443 -tls1_1` (должен упасть) и `-tls1_3` (должен пройти) | Только 443 доступен снаружи, старые протоколы TLS отключены |
| PID 1 — реальный сервис, не заглушка | `docker top mariadb` | Видно `mysqld` как PID 1, а не `bash`/`tail` |
| Второй пользователь БД существует, admin без "admin" в имени | `docker exec -it mariadb mariadb -u root -p... -e "SELECT User FROM mysql.user;"` | Оба обязательных пользователя видны |
